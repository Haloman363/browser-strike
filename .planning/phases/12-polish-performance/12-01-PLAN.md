# Phase 12: Polish & Performance - Plan 01: Core Optimizations

**Objective:** Reduce GPU overhead by optimizing compute dispatches, implement object instancing for common props, and ensure clean memory disposal.

## Key Goals
1. **Dynamic Compute Dispatches**: Stop calling `renderer.compute()` in `FXSystem.js` when no particles are active.
2. **Prop Instancing**: Implement `InstancedMesh` for crates in `Factory.js` to reduce draw calls.
3. **Memory Cleanup**: Implement explicit disposal of Three.js objects in `main.js` and `Maps_v2.js`.
4. **Weapon Smoothing**: Add basic interpolation to recoil recovery for smoother visuals.

## Proposed Changes

### `src/systems/FXSystem.js`
- Add `activeSparks`, `activeSmoke`, `activeFire` timers.
- When spawning, reset the timer to the maximum particle life.
- In `update()`, decrement timers and only call `renderer.compute()` if timer > 0.

### `src/Factory.js`
- Create a `CrateInstancedMesh` managed by the system or a static variable.
- Update `createCrate()` to optionally return an instance if the size matches standard (e.g., 20, 30, 40).
- *Alternative*: Refactor `Maps_v2.js` to collect crate positions and then create one `InstancedMesh` per size. (Better for performance).

### `main.js`
- Refactor the scene clearing loop to call `.dispose()` on `geometry` and `material` of every removed mesh.

### `src/systems/WeaponSystem.js`
- Smooth out the `recoil` and `cameraRecoil` decay in the `update()` loop using lerp or a more consistent exponential decay.

## Verification Plan
1. **Performance Test**: Compare FPS before and after changes (especially with no active particles).
2. **Memory Test**: Monitor heap size during repeated map reloads in Chrome DevTools.
3. **Visual Test**: Ensure instanced crates look identical to unique ones and maintain destructibility.
